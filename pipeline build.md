##### **Knowledge base creation :**the 'answer' column of the MIRIAD can be used but with a important condition; the LLM used to generate the final output should not be a truth generator it should only rephrase and align the retrieved content to the users' query.to use the 'answer' column as the knowledge source the final LLMs role should be locked as follows:it is allowed to, Rephrase, Combine multiple retrieved answers, Remove redundancy and Match user phrasing but is cannot be allowed to add Medical facts, infer beyond the text, generalize, explain mechanisms that are not stated.created a FAISS index using the 'answer' column and a Metadata file. the FAISS id of a vector is mapped with rows of the metadata file. FAISS index position == metadata list position.##### ##### **Scoring Module (Novel Component)**consist of 2 SLMs. both work as filters. begins the second filtration phase after the hybrid retrieval phase(uses the top-k results returned by hybrid retrieval).###### **SLM 1- Gatekeeper (Constraint-aware filtration):**it takes the user query and extracts certain constraints from the user query. basically it decides what things the 'perfect' answer should have to a given user query. based on those decided constraints it filters the initial result set. SLM 1 is trained to identify the non-negotiables a certain chunk should have in-order for it to be a suitable answer for the user query.*ex: user query -> constraints (things a correct answer should mention).***Process of creating SLM 1:****Step 1: Creating the fine-tuning dataset*** loaded the entire neurology and neurosurgery dataset. found there was a class imbalance (a lot more neurology related rows than neurosurgery). sampled 'neurology' rows and got 7K from both specialties to a balanced dataset(there were only 7K of neurosurgery).* defined the constraint schema-which the model will learn to extract from the user query. then defined the teacher prompt that will be later used by 'gpt-4o-mini' when creating the fine-tuning dataset. the teacher prompt was specifically instructed to only extract constraints from the user query-to prevent data leakage.* the balanced 14K dataset had all matching samples. the question and it's matching answer chunk. we cannot properly train a 'Gatekeeper' by exposing it just to correct samples. we need pairs with incorrect 'question : answer' pairs as well. so i created 2x more negative pairs (pairing question with someone else's answer). negative pairs are more informative to a model during training than positive ones. the candidate pair file now had nearly 42K rows (with real and mismatched pairs).* now what left was the labelling task(training dataset creation). first the function that formats the teacher prompt was created. it puts the actual values to the defined prompt. this formatted prompt will be given to LLM (gpt-4o-mini)each time to do the labelling.* before labelling the entire dataset a small test was done. to check if the LLM does the labelling the way we need it to. just for this test, we included both the 'label' and the 'pair\_type' in the result to compare them later('label' is what's given for each question and answer pair by gpt-4o-mini. pair\_type is it's actual label.-label that is given for each row in the candidate pair dataset).* then the labelling task was conducted on the entire dataset (just 19K rows). my goal here was to extract the constraint schema and reasoning for each 'question : answer' pair on why they match or not (why their label is 'PASS' or 'FAIL'-the label was also given by the gpt-4o-mini but extracting the mentioned parts was the key need. LLMs label depends on the constraints). when I created the candidate dataset that part was not there. and I'm not a medical expert to do that manually and the volume of question answer pairs makes it difficult to do manually.* after labeling was done I created the final fine-tuning dataset. when fine-tuning a LLM there is a certain structure that the input data should be in to train it properly. so i converted each and every row in the labelled dataset into that format. the structure I used here is OpenAI's chat fine-tuning format.**Step 2: fine-tuning the SLM*** Llama-3.2-3B-Instruct was fine-tuned for this task as it is designed to follow specific instruction based tasks and good at JSON output. Also instruct models are good if the task involves specialized terms that are not commonly found in day-to-day language.* first I setup the environment. Downloaded the tokenizer and the quantized model weights.* LoRA was used to fine-tune. configured LoRA with the target modules(layers of the original model which will be getting adapters). 4 Attention layers (these pick up the task-specific attention patterns) and 3 MLP layers (to pick-up factual knowledge and domain specific reasoning) were included under target modules. next trainable adapter modules were attached to the mentioned layers (original weights are frozen only the adapters will be trained) - *model is now ready to start fine-tuning.** *before fine-tuning begins the dataset needs to be prepared*. 'tokenize\_chat' function is key here. it converts the rows of the fine-tuning dataset to text -> tokens. thee tokens are what the model understands. it also masks the non-assistant tokens of each input row, so the model does not compute loss over them. that means the loss will only be computed over what's given inside 'assistant' field. which means the model learns to correctly predict what's inside that field. which is:&nbsp;	*Assistant: {"constraints": {...}, "reasoning": "...", "label": "PASS"}*         *└─────────── ONLY THIS PART ──────────┘** then the  training configuration was set (optimization, evaluation, batch size, checkpointing).then the training was conducted.**Step 3: Evaluating the trained model*** loaded the LoRA adapter and attached those adapters to *...*	*After:*	*LLaMA Layer:*        *q\_proj: \[W\_original] + \[A × B]  ← LoRA adapter attached!*               *└─ frozen ─┘   └─ loaded from disk ─┘*        *k\_proj: \[W\_original] +
